Algorithms such as MD5, SHA-1, MD4, and SHA-0 are cryptographically broken and must not be used in security-sensitive contexts. They are vulnerable to collision and preimage attacks.

This rule covers:

  • JDK MessageDigest.getInstance()
  • Apache Commons DigestUtils.md5(), DigestUtils.sha1()
  • Guava Hashing.md5(), Hashing.sha1(), Hashing.crc32()
  • BouncyCastle new MD5Digest(), new SHA1Digest(), new MD4Digest()

Severity is adjusted by context: usage near security-related variable or method names (e.g. password, token, auth) raises the issue to CRITICAL. Usage clearly in a non-security context (e.g. cacheKey, etag) is skipped entirely.

Noncompliant Code Example

// Noncompliant — MD5 near security context
MessageDigest md = MessageDigest.getInstance("MD5"); // Noncompliant
byte[] tokenHash = md.digest(token.getBytes());

// Noncompliant — Apache Commons
String hash = DigestUtils.md5Hex(password); // Noncompliant

// Noncompliant — BouncyCastle
MD5Digest digest = new MD5Digest(); // Noncompliant

Compliant Solution

// Use SHA-256 or stronger for general hashing
MessageDigest md = MessageDigest.getInstance("SHA-256");

// Use a password hashing function for passwords
// See crypto:WeakPasswordHash for password-specific guidance
